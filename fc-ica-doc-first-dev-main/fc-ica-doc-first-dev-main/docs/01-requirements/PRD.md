# 제품 요구사항 문서 (PRD)

## 프로젝트명: TaskFlow — 태스크 관리 SaaS

---

## 1. 프로젝트 개요

### 1.1 배경

소규모 팀(3~15명)이 프로젝트 단위로 업무를 관리할 때, 기존 도구(Jira, Asana 등)는 기능이 과도하게 복잡하거나 가격이 부담스러운 경우가 많다. TaskFlow는 **핵심 기능에 집중한 경량 태스크 관리 SaaS**를 제공하여, 스타트업과 소규모 팀이 빠르게 도입하고 즉시 생산성을 높일 수 있도록 한다.

### 1.2 목표

| 구분 | 목표 |
|------|------|
| 단기 (3개월) | MVP 출시 — 사용자 인증, 프로젝트/태스크 CRUD, 칸반 보드뷰 |
| 중기 (6개월) | 유료 전환율 5% 달성, MAU 1,000명 |
| 장기 (12개월) | 팀 협업 기능 강화(댓글, 파일 첨부, 알림) 및 MAU 5,000명 |

### 1.3 성공 지표 (KPI)

| 지표 | 목표값 | 측정 방법 |
|------|--------|-----------|
| 회원가입 전환율 | 랜딩페이지 방문자 대비 15% | Analytics |
| 주간 활성 사용자(WAU) | 출시 후 1개월 내 200명 | 로그인 기반 집계 |
| 태스크 생성 수 | 사용자당 주 10건 이상 | DB 집계 |
| 평균 세션 시간 | 8분 이상 | Analytics |
| NPS | 40 이상 | 인앱 설문 |

## 2. 사용자 분석

### 2.1 타겟 사용자

| 페르소나 | 설명 |
|----------|------|
| **팀 리더 (김수진, 32세)** | 스타트업 PO. 5명 팀의 스프린트를 관리하며, 각 멤버의 진행 상황을 한눈에 파악하고 싶다. |
| **개발자 (이정호, 27세)** | 백엔드 개발자. 본인에게 할당된 태스크를 빠르게 확인하고 상태를 변경하고 싶다. |
| **디자이너 (박민아, 29세)** | UI/UX 디자이너. 태스크에 첨부된 디자인 파일을 확인하고 피드백을 주고 싶다. |

### 2.2 사용자 니즈

1. **직관적인 보드뷰**: 드래그앤드롭으로 태스크 상태를 빠르게 변경
2. **빠른 태스크 생성**: 최소한의 입력으로 태스크를 즉시 생성
3. **프로젝트별 분리**: 여러 프로젝트를 독립적으로 관리
4. **팀 협업**: 멤버 초대, 태스크 담당자 지정
5. **실시간 현황 파악**: 대시보드에서 전체 프로젝트 상태 요약

### 2.3 사용 시나리오

#### 시나리오 1: 팀 리더가 새 프로젝트를 시작하는 경우

1. 로그인 후 대시보드에서 "새 프로젝트" 버튼 클릭
2. 프로젝트명, 설명 입력 후 생성
3. 팀 멤버를 이메일로 초대
4. 칸반 보드에서 태스크 생성 및 담당자 지정
5. 보드뷰에서 전체 진행 상황 모니터링

#### 시나리오 2: 개발자가 태스크를 처리하는 경우

1. 로그인 후 대시보드에서 할당된 프로젝트 선택
2. 칸반 보드에서 본인 태스크 확인
3. "To Do" → "In Progress"로 드래그
4. 작업 완료 후 "In Progress" → "Done"으로 드래그

## 3. 기능 요구사항

### 3.1 MVP 기능 (Phase 1~5)

#### FR-01: 사용자 인증

| ID | 기능 | 설명 |
|----|------|------|
| FR-01-01 | 회원가입 | 이메일, 비밀번호, 이름으로 회원가입 |
| FR-01-02 | 로그인 | 이메일/비밀번호 로그인, JWT 토큰 발급 |
| FR-01-03 | 로그아웃 | 클라이언트 측 토큰 삭제 |
| FR-01-04 | 토큰 갱신 | Access Token 만료 시 Refresh Token으로 갱신 |

#### FR-02: 프로젝트 관리

| ID | 기능 | 설명 |
|----|------|------|
| FR-02-01 | 프로젝트 생성 | 이름, 설명을 입력하여 프로젝트 생성 |
| FR-02-02 | 프로젝트 목록 조회 | 사용자가 속한 프로젝트 목록 표시 |
| FR-02-03 | 프로젝트 상세 조회 | 프로젝트 정보 및 멤버 목록 표시 |
| FR-02-04 | 프로젝트 수정 | 프로젝트명, 설명 수정 (소유자만) |
| FR-02-05 | 프로젝트 삭제 | 프로젝트 삭제 (소유자만, 소프트 삭제) |
| FR-02-06 | 멤버 초대 | 이메일로 기존 사용자를 프로젝트에 초대 |

#### FR-03: 태스크 관리

| ID | 기능 | 설명 |
|----|------|------|
| FR-03-01 | 태스크 생성 | 제목, 설명, 상태, 우선순위, 담당자 지정 |
| FR-03-02 | 태스크 목록 조회 | 프로젝트 내 태스크 목록 (필터, 정렬 지원) |
| FR-03-03 | 태스크 상세 조회 | 태스크 상세 정보 표시 |
| FR-03-04 | 태스크 수정 | 태스크 정보 수정 |
| FR-03-05 | 태스크 삭제 | 태스크 삭제 (소프트 삭제) |
| FR-03-06 | 상태 변경 | TODO → IN_PROGRESS → DONE 상태 전환 |

#### FR-04: 칸반 보드뷰

| ID | 기능 | 설명 |
|----|------|------|
| FR-04-01 | 보드 렌더링 | 상태별 컬럼(TODO, IN_PROGRESS, DONE)으로 태스크 표시 |
| FR-04-02 | 드래그앤드롭 | 태스크 카드를 다른 컬럼으로 드래그하여 상태 변경 |
| FR-04-03 | 카드 미리보기 | 카드에 제목, 우선순위, 담당자 아바타 표시 |

### 3.2 선택 기능 (Post-MVP)

| ID | 기능 | 우선순위 |
|----|------|----------|
| FR-05-01 | 리스트뷰 | Medium |
| FR-05-02 | 태스크 의존성 (선행 태스크 지정) | Low |
| FR-05-03 | 파일 첨부 (S3 연동) | Medium |
| FR-05-04 | 태스크 댓글 | Medium |
| FR-05-05 | 실시간 알림 (WebSocket) | Low |
| FR-05-06 | 검색 기능 | Medium |

## 4. 비기능 요구사항

### 4.1 성능

| 항목 | 요구사항 |
|------|----------|
| API 응답 시간 | 95th percentile 기준 200ms 이내 |
| 페이지 로드 시간 | LCP(Largest Contentful Paint) 2.5초 이내 |
| 동시 접속자 | 최소 100명 동시 접속 지원 |
| 데이터베이스 | 프로젝트당 태스크 10,000건까지 성능 저하 없음 |

### 4.2 보안

| 항목 | 요구사항 |
|------|----------|
| 인증 | JWT 기반 Bearer Token 인증 |
| 비밀번호 | bcrypt 해싱 (최소 12 rounds) |
| HTTPS | 프로덕션 환경에서 TLS 1.2 이상 필수 |
| 입력 검증 | 모든 API 입력에 대해 서버 측 유효성 검증 |
| CORS | 허용된 도메인만 접근 가능 |
| SQL Injection | ORM 사용 및 파라미터 바인딩 |

### 4.3 접근성

| 항목 | 요구사항 |
|------|----------|
| WCAG | 2.1 AA 수준 준수 |
| 키보드 네비게이션 | 모든 주요 기능 키보드로 조작 가능 |
| 스크린 리더 | 주요 UI 요소에 ARIA 레이블 제공 |
| 색상 대비 | 텍스트 대비율 4.5:1 이상 |
